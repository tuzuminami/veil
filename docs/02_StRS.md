# 02. StRS — ステークホルダー要求仕様
**参照:** ISO/IEC/IEEE 29148。

## 1. ステークホルダー要求
| ID | 要求 |
|---|---|
| STR-VEIL-001 | 開発者はDocker Composeのみでローカル実行、seedデータ投入、主要APIの確認を行えること。 |
| STR-VEIL-002 | プロダクト責任者は、設定・Version・状態変更がいつ、誰により、どの理由で行われたかを追跡できること。 |
| STR-VEIL-003 | 運用者は、外部依存の障害時に安全な縮退動作と復旧対象を識別できること。 |
| STR-VEIL-004 | セキュリティ担当者は、tenant境界、認可、Secret非露出、監査ログのテスト証跡を確認できること。 |
| STR-VEIL-005 | OSS利用者は、安定したPlugin SPIを通じて外部Providerや独自ルールを差し込めること。 |
| STR-VEIL-006 | 商用利用者は、ソースを改変・再配布・ホスティングしても、コアライセンスで不当に制約されないこと。 |

## 2. 利用シナリオ
### UC-01: 開発者が主要機能を試す
1. 開発者は`docker compose up`を実行する。
2. seed用APIキーでサンプルtenantを選択する。
3. OpenAPIに従って主要リソースを作成する。
4. 監査ログとメトリクスで結果を確認する。

**成功条件:** 15分以内に主要フローが再現できる。

### UC-02: 運用者が設定変更を安全に公開する
1. 新Versionを作成する。
2. static validationとdry-runを実行する。
3. レビュー承認後に公開する。
4. 既存Sessionまたは既存データへの適用範囲を確認する。

**成功条件:** 既存の不変Versionを上書きせず、rollbackまたは参照先切替ができる。

### UC-03: 外部依存が失敗する
1. AdapterまたはPluginがtimeoutになる。
2. システムは失敗原因を型付きエラーと監査イベントに記録する。
3. 構成に従って安全な停止、保留、または機能限定を行う。

**成功条件:** 認可・ポリシー・データ境界を暗黙に緩めない。
