# 04. AD — アーキテクチャ記述
**参照:** ISO/IEC/IEEE 42010。

## 1. アーキテクチャ上の関心事
- Tenant分離と認可境界
- 外部Provider / Pluginへの依存制御
- Version不変性と再現性
- 監査、観測性、障害時の安全な縮退
- OSSとしてのローカル実行性、商用組込みやすさ

## 2. コンテキストビュー
```text
Client / SDK / CLI
        │ HTTPS + OAuth/API key
        ▼
    VEIL API ────────► AuthN/AuthZ Adapter
        │
        ├──────────────► Plugin / Provider Adapters
        │
        ├──────────────► PostgreSQL
        │
        └──────────────► Metrics / Traces / Audit Sink
```

## 3. コンテナビュー
| コンテナ | 責務 | 禁止事項 |
|---|---|---|
| API | HTTP、schema検証、認可、DTO変換 | domainロジックやDB直書き |
| Core | ユースケース、状態遷移、ポリシー呼出し | HTTP/ORM/SDKへの直接依存 |
| Storage Adapter | 永続化、トランザクション、migration | 認可判断 |
| Plugin Host | SPIロード、capability検証、timeout制御 | コア状態の直接更新 |
| Worker | 再試行可能な非同期処理 | 同期APIの代替 |
| SDK/CLI | 開発者体験、API利用 | 事業ルールの複製 |

## 4. 情報ビュー
- 全リソースはtenant境界を持つ。
- Version化対象は**公開後不変**とする。
- 監査イベントはappend-onlyとし、実データ最小化を前提にする。
- 外部入力はJSON Schemaで検証し、domainへ不正形状を渡さない。

## 5. 開発ビュー
```text
packages/core        ← domain / use cases / ports
packages/contracts   ← OpenAPI / JSON Schema / generated types
packages/plugins     ← SPI / built-in adapters
apps/api             ← transport / auth / DI
packages/sdk-ts      ← client SDK
tests                ← test fixtures and acceptance suites
```

## 6. デプロイメントビュー
- 開発: Docker Compose（API、PostgreSQL、必要なmock adapter）。
- 本番: stateless APIを水平スケール。Workerは別Deployment。DB migrationは単一jobで実施。
- Secret: KMS / Vault / 環境変数Adapterを利用し、アプリDBに平文保存しない。

## 7. 主要設計判断
| ADR | 判断 | 理由 |
|---|---|---|
| ADR-001 | Hexagonal Architecture | Provider・DB・認証の差替を容易にする。 |
| ADR-002 | PostgreSQLを基準DB | トランザクション、JSON、監査、運用容易性を両立する。 |
| ADR-003 | JSON Schema at boundary | 早期検証とSDK生成の一貫性を確保する。 |
| ADR-004 | Append-only audit | 重要操作の説明可能性と改ざん耐性を高める。 |
| ADR-005 | Pluginの明示登録 | 暗黙ロードを避け、攻撃面と障害範囲を縮める。 |
