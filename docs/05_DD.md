# 05. DD — 設計記述
    **参照:** ISO/IEC/IEEE 12207 の実装・検証・構成管理プロセスに沿った設計記述。

    ## 1. モジュール設計
    | モジュール | 公開する責務 | 主なテスト |
    |---|---|---|
    | `core/domain` | 不変条件、値オブジェクト、ドメインイベント | unit |
    | `core/application` | ユースケース、トランザクション境界、ports | unit / integration |
    | `adapters/storage` | PostgreSQL永続化、migration、tenant制約 | integration |
    | `adapters/plugins` | 外部Plugin、Provider、timeout、capability検証 | contract |
    | `apps/api` | HTTP、認可、入力検証、エラーマッピング | API / e2e |
    | `sdk-ts` | 型安全なクライアント、retryの安全な範囲 | contract |

    ## 2. 主要データモデル
    - `policybundle`: `PolicyBundle`を表す永続化または値オブジェクト。
- `policyversion`: `PolicyVersion`を表す永続化または値オブジェクト。
- `decisionrequest`: `DecisionRequest`を表す永続化または値オブジェクト。
- `decision`: `Decision`を表す永続化または値オブジェクト。
- `evidence`: `Evidence`を表す永続化または値オブジェクト。
- `ageassuranceclaim`: `AgeAssuranceClaim`を表す永続化または値オブジェクト。
- `boundarydeclaration`: `BoundaryDeclaration`を表す永続化または値オブジェクト。
- `appeal`: `Appeal`を表す永続化または値オブジェクト。
- `auditevent`: `AuditEvent`を表す永続化または値オブジェクト。

    共通カラム: `id`, `tenant_id`, `created_at`, `created_by`, `updated_at`, `version`, `correlation_id`。

    ## 3. トランザクション規則
    1. 認可済みのtenant contextをトランザクション開始前に確立する。
    2. ドメイン不変条件を検証する。
    3. 主リソース、監査イベント、outbox eventを同一トランザクションで確定する。
    4. 外部I/Oはoutbox消費後に実行する。外部失敗をDB transactionへ巻き戻さない。
    5. 再送可能な変更はidempotency keyと結果を保存する。

    ## 4. エラー規約
    | 区分 | HTTP | 例 |
    |---|---:|---|
    | 入力不正 | 400/422 | Schema違反、値域外 |
    | 未認証 | 401 | token不在・無効 |
    | 権限不足 | 403 | tenant越境、scope不足 |
    | 競合 | 409 | Version衝突、公開済み不変リソース更新 |
    | 依存障害 | 503 | Adapter timeout、必須Plugin停止 |
    | 安全停止 | 409/423/503 | ポリシー未解決、fail-closed |

    ## 5. Migrationと互換性
    - migrationは前方互換を基本にし、expand → deploy → backfill → contractの順で行う。
    - Version化データの既存行を上書きしない。
    - API、Schema、Plugin SPIの互換性は個別にSemVer管理する。
